# Codex Designer

Codex Designer is a local Electron app that turns the Codex workflow into a structured, workspace-based process:

- **Planning** via guided Q&A
- **Implementation** with resumeable Codex runs
- **Testing** with manual acceptance tracking and feedback loops

The app is designed for people who want stronger continuity between model sessions and repository-level work.

## What this repo contains

- `src/` — Vue 3 renderer (UI, run state, prompts, session workbench).
- `electron/` — Electron main process, preload bridge, and orchestration logic.
- `docs/` — local planning artifacts generated by the app and implementation notes.
- `build/` and `public/` — Electron/Vite packaging assets.

## Core capabilities

- Pick or open a local workspace (repo).
- Work with feature sessions based on `docs/<slug>.plan.md`:
  - Plan notes (`*.plan.md`)
  - Q&A thread (`*.qna.md`)
  - Implementation log (`*.impl.md`)
  - Test plan JSON + markdown (`*.test.json`, `*.test.md`)
- Run Codex in multiple modes (planning, implementation, testing, generic) with per-role profiles.
- Configure model + reasoning effort per role and per run.
- Stream run events in real time with run history persistence.
- Attach local images to prompts and include them in markdown artifacts.
- Basic Git tooling:
  - status / diff / diff-stat
  - fetch / pull / push
  - branch list / switch / create / merge
- Workspace-level persistence:
  - `.codex-designer/share/` for shareable workspace config and profiles
  - `.codex-designer/cache/` for local scratch state (always ignored)

## Prerequisites

- Node.js 18+ (current Electron + Vite toolchain expects a modern Node runtime).
- npm.
- Git on `PATH`.
- A working Codex CLI binary compatible with the installed `@openai/codex-sdk` package.

> The app does not perform Codex authentication itself. Use your normal Codex setup in your environment and make sure `codex app-server` (or equivalent `codex` behavior) is available through the SDK-bundled runtime.

## Get started

```bash
cd /path/to/codex-designer
npm install
npm run dev
```

The app runs in desktop mode via Vite and Electron.

## Useful scripts

- `npm run dev` — start renderer + Electron dev flow.
- `npm run build` — type-check and build the renderer bundle.
- `npm run preview` — serve the built renderer bundle.
- `npm run pack` — build and package with `scripts/pack.mjs`.
- `npm run dist` — create a distributable via Electron Builder.

## Runtime flow (high level)

1. Renderer uses `window.codexDesigner` (exposed by `electron/preload.ts`) for all privileged actions.
2. Main process IPC handlers in `electron/ipc.ts` execute filesystem, git, and run orchestration tasks.
3. Codex runtime orchestration lives in `electron/lib/codexRuns.ts` and streams JSONL events to the UI.
4. Run metadata and event logs are stored in Electron user data directory (`<userData>/codex-designer/runs`), while workspace metadata and generated artifacts are kept inside the workspace itself.

## Data stored by the app

### In-workspace

| Path | Purpose |
| --- | --- |
| `docs/*.plan.md` | Generated/maintained implementation plan |
| `docs/*.qna.md` | Structured Q&A log |
| `docs/*.impl.md` | Implementation notes / progress |
| `docs/*.test.json` | Structured test results/history |
| `docs/*.test.md` | Rendered test notes |
| `docs/assets/<slug>/` | Prompt images and pasted attachments |
| `.codex-designer/share/` | Shareable workspace config (profiles, optional house-style) |
| `.codex-designer/cache/` | Internal cache/state (`state.json`) |

### Electron app data

- Recent workspaces and UI state: `~/.config` / app user data equivalent location (platform-specific).
- Run logs: `<userData>/codex-designer/runs/*`.

## Model list and session controls

- The model dropdowns call into the app’s model inspector path (`model/list`) from the configured Codex runtime.
- Model catalog is treated as dynamic:
  - it is cached in memory while the session is active
  - stale cache is refreshed automatically
  - can be refreshed manually

## Security posture

- No hard-coded secrets are committed in this repo.
- The app does not manage API keys directly.
- Credentials and auth are intentionally delegated to your local Codex configuration.
- Local state is separated between:
  - repository-local workspace files (`docs`, `.codex-designer/share`, `.codex-designer/cache`)
  - app-local machine state (`userData`).
- Keep `docs/.codex-designer` shareability in mind before committing:
  - `.codex-designer/cache` is always non-shareable
  - `.codex-designer/share` is configurable per workspace

See `SECURITY.md` for the full policy and reporting guidance.

## Development expectations

- Run through normal TypeScript and Vue workflows in `src/` before changing core IPC behavior in `electron/`.
- Preserve the renderer/main process boundary: filesystem and shell access should remain in the main process and be forwarded via explicit IPC.
- Keep all persistence paths deterministic and sandboxed with validation helpers (`safePath`).
- Use the existing structured prompt builders in `src/lib` and `electron/lib` instead of ad-hoc string-heavy logic.

## Known limitations / current scope

- Early-stage app intended for desktop users on macOS/Windows/Linux.
- Run/profile behavior currently assumes a Codex-capable environment.
- Local markdown rendering is sanitized (`DOMPurify`) before injection.
- The UI is opinionated around a right-side docs panel and split workbench flow.
